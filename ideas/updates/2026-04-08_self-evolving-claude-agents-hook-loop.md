# Self-Evolving Claude Agents via a 4-Hook Run/Log/Improve Loop

## What Is This

A concrete architecture for making Claude Code agents learn from their own mistakes across sessions — without any manual prompt editing. Four hook scripts silently observe every run, accumulate structured observations, and periodically trigger a background "dream worker" that reads recent runs, detects patterns (e.g., 3 identical failures from the same agent), and writes rules back into `.claude/rules/` or agent-specific markdown files. The next run, the agent already knows what broke before.

## Why It Matters

Claude Code has no cross-session memory by default — the same agent will repeat the same failure indefinitely. This pattern closes that loop permanently using only hooks (which are natively supported) and a background script. No external infra required. It's also drop-in: the hook files work in any Claude Code project immediately.

## How To Use It

**The 4-hook architecture:**

1. **`SessionStart` hook** — Fires when Claude opens. Writes a "state bridge" file for the current session (session ID, timestamp, context). This is the scaffold all other hooks append to.

2. **`PostToolUse` hook** — Fires after every tool call. Appends to the state bridge: which tool ran, which skills were read, any error output. Creates a compact per-tool breadcrumb trail cheaper than parsing full transcripts.

3. **`Stop` hook** — Fires when Claude finishes responding. Writes a full observation entry to a persistent log file: agent name, success/fail, duration, output preview. This is the "one record per run" that the dream worker reads.

4. **Gate hook (also `Stop` or `SessionEnd`)** — Checks two conditions before spawning the dream worker:
   - Has it been at least 4 hours since last analysis?
   - Are there at least 3 new observations logged since last run?
   - If both true → spawn the dream worker as a background process.

**The dream worker:**
- Reads the last 20 run observations from the log
- Looks for concrete patterns: same agent, same step, same failure occurring 3+ times
- Spawns 4 sub-agent reviewers in parallel, each with a different lens:
  - Reviewer 01: spots failure patterns
  - Reviewer 02: finds slow/timeout spots
  - Reviewer 03: catches bad outputs / hallucinations
  - Reviewer 04: checks general quality
- 4 reviewers are used to cancel individual LLM bias — one reviewer's blind spots get caught by another
- Writes discovered rules to one of two paths:
  - `.claude/rules/` → global rules, loaded for all agents automatically on next session
  - `.claude/learning/agents/{agent-name}.md` → agent-specific rules (custom convention, not a built-in Claude Code path)

**Hook configuration:**
Hooks must be registered in `.claude/settings.json` — they do NOT auto-fire just by placing scripts in `.claude/hooks/`. The scripts can live in `.claude/hooks/` by convention, but you must point to them in settings:

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "hooks": [{ "type": "command", "command": ".claude/hooks/posttool-use.js" }]
      }
    ],
    "Stop": [
      {
        "hooks": [{ "type": "command", "command": ".claude/hooks/on-stop.js" }]
      }
    ]
  }
}
```

**Rule injection:**
Rules written to `.claude/rules/*.md` are automatically loaded by Claude Code on the `InstructionsLoaded` event (verified in docs). This is the key mechanism that feeds learned rules back into the agent's context on the next run.
